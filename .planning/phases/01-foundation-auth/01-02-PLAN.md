Create `app/db/session.py`:
- Imports: `from sqlalchemy.ext.asyncio import AsyncSession, create_async_engine, async_sessionmaker`
- Imports: `from app.core.config import get_settings`
- Function `get_async_engine()` — calls `get_settings()`, returns `create_async_engine(settings.DATABASE_URL, echo=False, pool_pre_ping=True)`
- Module-level: `AsyncSessionFactory = async_sessionmaker(get_async_engine(), class_=AsyncSession, expire_on_commit=False)` — NOTE: this eager init will fail without DATABASE_URL; gate behind a lazy factory pattern instead
- Better pattern: `_engine = None; _session_factory = None` — lazy init via `def _get_session_factory()` that creates engine once
- Async generator `get_session()`:
  ```python
  async def get_session() -> AsyncSession:
      async with _get_session_factory()() as session:
          yield session
  ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| env→Settings | DATABASE_URL and JWT_SECRET_KEY arrive from .env; missing values must fail fast at startup |
| ORM→DB | SQLAlchemy ORM maps Python objects to PostgreSQL; schema must enforce constraints at DB level not just app level |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-02-01 | Information Disclosure | app/core/config.py | high | mitigate | Settings class uses required fields with no default for JWT_SECRET_KEY and DATABASE_URL — missing values raise ValidationError at startup, not silently using insecure fallbacks |
| T-01-02-02 | Tampering | app/db/models.py User.email | medium | mitigate | DB UNIQUE constraint on users.email enforced at PostgreSQL level (not just app) — catches race-condition duplicate registration; IntegrityError maps to 409 in auth service |
| T-01-02-03 | Elevation of Privilege | app/db/models.py Company.ticker | low | accept | ticker is VARCHAR(20) PK; uppercase normalisation enforced at service layer only — a ticker with wrong case could theoretically create a duplicate row but no user-facing privilege escalation possible; normalisation guard added in auth service (plan 01-04) |
| T-01-02-04 | Repudiation | app/db/models.py soft-delete | low | accept | ResearchMemo has deleted_at (soft delete); hard deletes are prohibited by design; no audit trail beyond this in Phase 1 — acceptable for walking skeleton |
</threat_model>
