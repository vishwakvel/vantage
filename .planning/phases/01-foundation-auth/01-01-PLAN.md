<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer→docker-compose | Dev pulls external Docker images; image provenance must be pinned |
| build→pip install | pip installs packages from PyPI at build time; malicious packages would run in the container |
| container→host | .env file with secrets is bind-mounted; secrets must never enter the image layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-P1-01 | Tampering | requirements/base.txt | high | mitigate | Pin all packages to exact versions (==) in requirements/base.txt to prevent dependency confusion or supply-chain substitution |
| T-01-P1-02 | Information Disclosure | Dockerfile | high | mitigate | COPY only app source; never COPY .env into image; all secrets sourced from env_file at runtime, not baked into image layers |
| T-01-P1-03 | Elevation of Privilege | docker-compose.yml | medium | mitigate | PostgreSQL/Redis use non-default credentials defined in .env; .env is gitignored; .env.example has no real secrets |
| T-01-P1-04 | Denial of Service | api service startup | low | accept | No rate limiting on health endpoint; acceptable for internal-only dev environment; Milestone 2 adds production hardening |
| T-01-P1-SC | Tampering | pip installs (requirements/base.txt) | high | mitigate | All packages are established PyPI packages (fastapi, uvicorn, sqlalchemy, asyncpg, alembic, pydantic, python-jose, passlib, redis, httpx) verified by package name and version |
</threat_model>
